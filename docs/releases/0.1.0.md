# Adaptive Learning Agent 0.1.0 — Runtime Proof

Release date: 2026-07-18
Status: pre-alpha runtime and architecture proof

## Purpose

Version 0.1.0 demonstrates that deterministic learning behavior can remain in a small runtime-independent Python core while Hermes supplies the conversation. It is a technical baseline for later reviewed subject packs, not a certification-preparation product.

## Implemented

- Standard-library Python core for strict pack validation/install, deterministic question selection and exact scoring, confidence persistence, restart-safe sessions, progress counts, and challenge quarantine.
- Eight-table SQLite schema with foreign keys, immediate write transactions, one-active-session enforcement, and immutable attempts at the application boundary.
- Exactly ten JSON-compatible contract operations and a thin project-local Hermes adapter exposing exactly ten `ala_*` tools.
- Strict format-0.1 unpacked JSON/Markdown pack with canonical digest and controlled local installation.
- Synthetic `fixture-basics` pack and one concise Hermes workflow skill.

## Boundaries proven

- The core and contract import no Hermes, MCP, model-provider, or network client.
- Hermes owns conversation and presentation; Python owns learning decisions and state transitions.
- SQLite, not conversation or agent memory, is authoritative learner state.
- The adapter delegates through `ApplicationService` and `ToolContract`; it does not import SQLite/storage internals or contain subject-specific behavior.
- Pack input remains portable files outside both Hermes and the database.

## Tests and CI

The final installed-package suite passed 29/29 tests on CPython 3.12.13, 3.13.14, and 3.14.6 with equivalent tests and no package/test warnings. Hosted [GitHub Actions run 29665581746](https://github.com/amcclure1/adaptive-learning-agent/actions/runs/29665581746) passed all three matrix jobs. Diagnostic core statement coverage was 87% (614 statements, 81 missed); every accepted AT-01 through AT-12 behavior has a direct test.

Run locally from the repository root:

```powershell
$env:PYTHONPATH='src'
python -m unittest discover -s tests -v
```

## Hermes result

Pinned Hermes v0.18.2 was verified on Windows using an isolated `adaptive-learning-dev` CLI/profile. Real Codex-backed checks covered health, initialization, validation/install, exact question presentation, deterministic feedback, process-exit/restart reconstruction in a fresh conversation, challenge quarantine, finish, and rejection of a post-completion resubmit without changing immutable attempts.

This release does not claim compatibility with Linux, macOS, Hermes Desktop, the gateway, profile export/import, aliases, another Hermes version, or another runtime.

## Launch

The isolated profile and pinned Hermes installation described in the integration handoff must already exist. Launch from this trusted repository root:

```powershell
$env:HERMES_ENABLE_PROJECT_PLUGINS='1'
& "$env:USERPROFILE\.local\bin\hermes.exe" -p adaptive-learning-dev --skills adaptive-learning:adaptive-learning
Remove-Item Env:HERMES_ENABLE_PROJECT_PLUGINS
```

Project plugins execute repository code. Keep the discovery gate process-local, launch only from a trusted checkout, and do not use the synthetic fixture as study advice.

## Limitations

- One local learner and at most one active session; no multi-user hosting or concurrent-process guarantee.
- Practice only; no mastery, scheduling, readiness, adaptive sequencing, or exam simulation.
- Synthetic fixture only; no real subject pack, subject builder, evidence workflow, or content-review administration.
- No application backup/restore, encryption at rest, or hostile same-account-process isolation.
- Project-local Hermes integration requires a trusted current working directory and manual profile allow-list behavior.
- Pack-directory/database installation is not fully atomic across competing processes.

## Known issues

- In Hermes v0.18.2, `hermes plugins enable` does not manage a project-only plugin; the profile uses an explicit YAML allow-list and the launch-time discovery gate.
- `hermes config set` represented the plugin list as a string during setup; the isolated profile required a YAML sequence.
- Project-local skills are not auto-discovered, so the plugin registers the qualified skill and launch preloads it.
- The profile reports schema version 0 with a migration available; no unrelated migration was run.
- Validation of an unexpected existing database can create expected schema objects before it subsequently fails closed.
- Defensive malformed-input and local-failure branches account for most uncovered core lines.
