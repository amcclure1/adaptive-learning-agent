# Proposed JSON Tool Contract

Status: minimal contract `0.1` accepted; broader `ala.tools.v1` proposed
Design baseline: 2026-07-18

## Accepted version 0.1 contract

The normative first contract is the ten-tool JSON contract in [`mvp-vertical-slice.md`](mvp-vertical-slice.md): `system.health`, `learner.initialize`, `pack.validate`, `pack.install`, `study.start`, `study.next`, `study.submit`, `study.status`, `study.finish`, and `question.challenge`.

Mutation retries are handled through concrete uniqueness and payload-comparison rules. Version 0.1 has no generic request ID, idempotency key, audit tool, authoring tool, scheduler tool, backup tool, or archive tool. The broader `ala.tools.v1` proposal below is deferred and must not be implemented in the first slice.

## Deferred broader contract proposal

## Contract rules

- The same contract is used by in-process calls, the JSON CLI, and runtime adapters.
- Transport is not part of the contract. One JSON request produces one JSON response.
- Object keys use `snake_case`; identifiers are opaque strings.
- Unknown request fields are rejected in v1 unless a tool explicitly defines `options` for extensions.
- Mutations require `idempotency_key`; reads must omit it.
- Times are RFC 3339 UTC and are generated by the core, never trusted from the agent.
- The JSON channel contains no logs or prose outside the response envelope.
- Sensitive answer fields never appear in `ala_study_next`.

## Request envelope

```json
{
  "contract_version": "ala.tools.v1",
  "request_id": "01JZEXAMPLE000000000000000",
  "tool": "ala_study_submit",
  "idempotency_key": "01JZEXAMPLE-SUBMIT-00000001",
  "arguments": {
    "session_id": "01JZSESSION000000000000000",
    "presentation_id": "01JZPRESENT00000000000000",
    "response": {"option_ids": ["a"]},
    "confidence": 4
  }
}
```

`request_id` correlates logs and may be new on retry. `idempotency_key` identifies the logical mutation and must be reused on retry.

## Success envelope

```json
{
  "contract_version": "ala.tools.v1",
  "request_id": "01JZEXAMPLE000000000000000",
  "ok": true,
  "result": {},
  "meta": {
    "tool_version": "1.0.0",
    "state_changed": true,
    "replayed": false,
    "warnings": []
  }
}
```

## Error envelope

```json
{
  "contract_version": "ala.tools.v1",
  "request_id": "01JZEXAMPLE000000000000000",
  "ok": false,
  "error": {
    "type": "conflict",
    "code": "PRESENTATION_ALREADY_ANSWERED",
    "message": "This presentation already has an attempt.",
    "retryable": false,
    "details": {"presentation_id": "01JZPRESENT00000000000000"}
  },
  "meta": {
    "tool_version": "1.0.0",
    "state_changed": false,
    "replayed": false,
    "warnings": []
  }
}
```

Error types are `validation`, `not_found`, `conflict`, `policy`, `compatibility`, `storage`, and `internal`. Stable codes include `INVALID_ARGUMENT`, `PACK_INVALID`, `PACK_DIGEST_CONFLICT`, `PACK_UNAVAILABLE`, `EVIDENCE_GATE_FAILED`, `REVIEW_REQUIRED`, `SESSION_NOT_ACTIVE`, `NO_OUTSTANDING_PRESENTATION`, `PRESENTATION_ALREADY_ANSWERED`, `QUESTION_CHANGED`, `IDEMPOTENCY_CONFLICT`, `DATABASE_BUSY`, and `ADAPTER_INCOMPATIBLE`.

## Tool catalog

### System and learner

#### `ala_system_info` — read

Arguments: `{}`.

Returns contract version, application version, database schema version, supported pack versions, supported algorithm versions, and adapter information when available. It returns no credentials or filesystem secrets.

#### `ala_learner_upsert` — mutation

Arguments:

```json
{
  "learner_id": null,
  "display_name": "Alex",
  "timezone": "America/Chicago",
  "locale": "en-US"
}
```

`learner_id: null` creates a profile; a present ID updates allowed display fields. Returns the complete non-sensitive profile and `created: true|false`.

#### `ala_data_backup` — mutation

Arguments contain `destination_path`, optional `learner_ids` (empty means all local learner profiles), and `overwrite: false` by default. The core uses SQLite's online backup mechanism, adds a small JSON manifest with application/schema versions and file SHA-256, and returns those receipt fields. Pack files and Hermes data are not included.

#### `ala_data_restore` — mutation, human confirmation required

Arguments contain `source_path`, `expected_sha256`, `overwrite_existing: true|false`, and `human_confirmation: true`. Restore stages the database, verifies manifest compatibility, integrity, and foreign keys, then swaps it only while no application transaction is active. The adapter must not assert human confirmation on its own. On any failure, the live database remains unchanged.

### Packs

#### `ala_pack_list` — read

Arguments may filter by `status`, `pack_id`, or evidence mode. Returns installed pack receipts and compatibility/verification status, never answer keys.

#### `ala_pack_validate` — read

Arguments:

```json
{
  "source_path": "C:/packs/aws-sap-c02",
  "profile": "release",
  "check_urls": false
}
```

`profile` is `draft`, `release`, or `install`. `check_urls` is an explicit network-assistance request and defaults false. Returns parsed identity, computed digest, ordered diagnostics, counts, evidence summary, and `valid` boolean. Validation does not install or mutate the source.

#### `ala_pack_install` — mutation

Arguments contain `source_path` and optional `expected_sha256`. Returns the immutable install receipt. It always runs the `install` validation profile and rejects draft packs, unsafe archives, and version/digest conflicts.

#### `ala_pack_export` — mutation

Arguments:

```json
{
  "source_kind": "draft",
  "source_id": "01JZDRAFT0000000000000000",
  "destination_path": "C:/exports/aws-sap-c02-0.1.0.zip",
  "overwrite": false
}
```

`source_kind` is `draft` or `installed`. Release validation is mandatory. Returns destination, byte length, archive SHA-256, pack digest, and included-file manifest. It does not publish or upload.

### Authoring

Authoring calls operate only on a `draft_id`; arbitrary paths are not accepted after creation.

#### `ala_author_create` — mutation

Arguments provide title, proposed `pack_id`, proposed version, language, evidence mode, and an author attestation record. Returns `draft_id`, generated safe draft path, and initial diagnostics.

#### `ala_author_apply` — mutation

Applies one validated edit. Arguments:

```json
{
  "draft_id": "01JZDRAFT0000000000000000",
  "operation": "upsert_question",
  "expected_draft_digest": "<64-hex-digest>",
  "value": {}
}
```

Allowed operations are:

- `update_manifest`;
- `upsert_objective` / `remove_objective`;
- `upsert_source` / `remove_source`;
- `write_content` / `remove_content`;
- `upsert_claim` / `remove_claim`;
- `upsert_question` / `retire_question`.

The value is operation-specific and follows the pack format. The call writes atomically, invalidates reviews whose target digest changed, validates the draft, and returns the new digest plus diagnostics. It cannot write outside the draft or set `status: release`.

#### `ala_author_review` — mutation, human confirmation required

Arguments:

```json
{
  "draft_id": "01JZDRAFT0000000000000000",
  "target_type": "question",
  "target_id": "q-cross-account-private-access",
  "target_sha256": "<64-hex-digest>",
  "reviewer": {"id": "sam", "name": "Sam Reviewer"},
  "outcome": "accepted",
  "checks": {
    "answerability": "pass",
    "correctness": "pass",
    "objective_alignment": "pass",
    "evidence_support": "pass",
    "distractor_quality": "pass",
    "explanation_quality": "pass",
    "currency_scope": "pass",
    "rights_originality": "pass"
  },
  "notes": "Verified against the cited primary source.",
  "human_confirmation": true
}
```

The adapter must not call this tool with `human_confirmation: true` unless the current user explicitly supplied the outcome. The core rejects acceptance when any required check is not `pass`, the digest is stale, or independent-review labels conflict with policy.

#### `ala_author_status` — read

Returns draft identity/digest, validation diagnostics, objective/question/source counts, review coverage, expired items, and release readiness. It never changes status or exports.

### Study

#### `ala_study_start` — mutation

Arguments:

```json
{
  "learner_id": "01JZLEARNER00000000000000",
  "pack_id": "org.example.aws-sap-c02",
  "pack_version": "1.0.0",
  "goal": {"max_questions": 20, "objective_ids": []}
}
```

Returns the pinned session identity, pack digest, algorithm versions, goal, and initial summary. Starting does not present a question.

#### `ala_study_next` — mutation

Arguments contain `session_id`. Returns:

```json
{
  "session_id": "01JZSESSION000000000000000",
  "presentation_id": "01JZPRESENT00000000000000",
  "ordinal": 1,
  "question": {
    "question_id": "q-cross-account-private-access",
    "type": "single_choice",
    "prompt": "...",
    "options": [
      {"id": "a", "text": "..."},
      {"id": "b", "text": "..."}
    ],
    "response_schema": {"option_ids": "array[string]"},
    "objective_labels": ["Design solutions for new workloads"]
  },
  "selection_summary": {
    "reason": "due_low_mastery",
    "is_review": false
  }
}
```

It MUST NOT return `answer`, accepted forms, scoring tolerance beyond what the learner needs to format a response, distractor analysis, review records, or rationale. If a presentation is already outstanding, it returns that presentation rather than creating another.

#### `ala_study_submit` — mutation

Arguments contain `session_id`, `presentation_id`, a response object matching the question type, optional confidence 1-5, and optional measured `duration_ms`. Returns:

```json
{
  "attempt_id": "01JZATTEMPT00000000000000",
  "score_micros": 1000000,
  "is_correct": true,
  "accepted_answer": {"option_ids": ["a"]},
  "reviewed_rationale": "...",
  "evidence": [
    {
      "source_id": "aws-private-connect-docs",
      "title": "AWS PrivateLink concepts",
      "url": "https://docs.aws.amazon.com/vpc/latest/privatelink/concepts.html",
      "locator": "Concepts > access a service through an interface endpoint"
    }
  ],
  "progress_changes": [
    {
      "objective_id": "design-new-solutions",
      "before_micros": 0,
      "after_micros": 300000
    }
  ],
  "schedule": {
    "previous_box": 0,
    "new_box": 2,
    "due_at": "2026-07-21T20:15:00Z"
  }
}
```

The core derives all answer, score, rationale, evidence, progress, and schedule fields from the pinned pack and rules. Caller-supplied equivalents are invalid arguments.

#### `ala_study_status` — read

Arguments identify a learner and pack, with optional session ID. Returns mastery and coverage per objective, due counts, recent calibration, session totals, and readiness. It does not infer readiness from agent memory.

#### `ala_study_finish` — mutation

Arguments contain `session_id` and outcome `completed` or `abandoned`. Returns an immutable session summary. It refuses completion while an answer submission is in flight; an outstanding unanswered presentation may be marked skipped only when the caller explicitly requests it.

## Determinism and redaction

- Results are canonical JSON for hashing and idempotency.
- Lists with semantic order retain it; diagnostic and summary lists define stable sort keys.
- Read tools may return current timestamps only in `meta`; deterministic content uses stored values.
- Logs include request ID, tool, duration, result code, and entity IDs. They exclude OAuth material, answer keys, `accepted_answer`, raw response text, and rationale by default.
- Administrative tools may reveal pack answer keys only after explicit authoring/review intent; learner-mode tools follow the disclosure rules above.

## Compatibility

Adapters declare supported contract versions. Additive result fields may appear within `ala.tools.v1`; callers must ignore unknown result fields. Removing/renaming fields, changing scoring meaning, or adding required arguments requires a new contract major version. Algorithm versions are independent and are returned explicitly.
